3. Call `submitWizardAction(fullData)`
4. If result has `error`: set errorMessage to result.message, isSubmitting=false, STAY on step 2
5. If result has `disambiguationNeeded`: set politicians=result.politicians, actionResult=result, setStep(3)
6. If result has `success`: set actionResult=result, setStep(3)

**Error banner (moderation rejection per UI-SPEC):**
- Displayed in Step2Issue when errorMessage is set
- Full-width above textarea: `bg-airmail-rot/10 border-l-4 border-airmail-rot text-airmail-rot p-4 rounded-r-lg text-sm font-body`
- Auto-hides: pass errorMessage to Step2Issue, which clears it on textarea keystroke via a callback

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User browser to Client components | Form input entered by user in wizard steps |
| Client to Server Actions | WizardData submitted to submitWizardAction / selectPoliticianAction |
| Step2Issue to /api/transcribe | Audio blob from voice recording to transcription endpoint |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-16 | Spoofing | Step1Form | mitigate | Client-side Zod validation via react-hook-form prevents invalid PLZ/email from submission; server-side revalidation in submitWizardAction is the true security gate |
| T-02-17 | Tampering | Step2Issue textarea | mitigate | Input moderation (SAFE-01) via OpenAI Moderation API in server action before any AI processing |
| T-02-18 | Information Disclosure | WizardData in client state | accept | Data lives in React useState (browser memory only); no localStorage, no cookies, no persistence per D-16. Browser tab close discards all data. |
| T-02-19 | Denial of Service | Submit button | mitigate | isSubmitting flag disables button during server action call; prevents double submission |
</threat_model>
