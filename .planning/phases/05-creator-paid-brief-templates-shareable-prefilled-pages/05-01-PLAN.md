<objective>
Create the non-UI foundation for campaign mode: persistent campaign records, immutable revision snapshots, revocable creator tokens, and a server-side repository that later plans can call without duplicating data logic.

This plan must not touch the public landing or wizard UI yet. Success means later plans can create, activate, read, edit, pause, and archive campaigns through one clean backend contract.
</objective>

<must_haves>
- Campaign public content lives in Supabase, not in flat files or URL payloads.
- Every successful create/edit publish writes an immutable snapshot so history survives direct overwrite.
- Creator access links are revocable and expirable.
- Nothing in this plan stores visitor-generated letters or visitor-edited issue text.
</must_haves>
</context>

<tasks>

<task type="auto">
  <name>Task 1: Add the campaign tables and RLS baseline</name>
  <action>
    Add a Supabase migration that creates:

    - `campaigns`
    - `campaign_revisions`
    - `campaign_tokens`

    Required fields:

    - `campaigns`: `id`, `slug`, `creator_email`, `title`, `issue_text`, `description`, `creator_name`, `external_url`, `status`, `moderation_status`, `moderation_categories`, `stripe_checkout_session_id`, `stripe_payment_intent_id`, `paid_at`, `email_verified_at`, `activated_at`, `paused_at`, `archived_at`, `last_published_revision_id`, timestamps
    - `campaign_revisions`: `id`, `campaign_id`, `snapshot_reason`, public text fields snapshot, `moderation_status`, timestamps
    - `campaign_tokens`: `id`, `campaign_id`, `kind`, `token_hash`, `expires_at`, `used_at`, timestamps

    Add unique constraints for `campaigns.slug` and for active token hashes. Enable restrictive RLS and revoke anon/authenticated access. All writes happen through service-role code only.
  </action>
  <verify>
    <automated>rg -n "create table .*campaigns|create table .*campaign_revisions|create table .*campaign_tokens|enable row level security" web/supabase/migrations</automated>
  </verify>
</task>

<task type="auto">
  <name>Task 2: Implement typed repository helpers and status transitions</name>
  <action>
    Create `web/src/lib/campaigns/schema.ts` and `repository.ts` with:

    - typed campaign status and moderation status unions
    - slug normalization and validation
    - create/read/update helpers
    - publish-safe transition helpers (`markPaid`, `markEmailVerified`, `activateCampaign`, `pauseCampaign`, `archiveCampaign`)
    - snapshot creation helper that writes `campaign_revisions`

<task type="auto">
  <name>Task 3: Add revocable creator token and management session helpers</name>
  <action>
    Create `web/src/lib/campaigns/tokens.ts` and `session.ts`.

    Requirements:

    - tokens are random, hashed before storage, time-limited, and single-use
    - support at least two kinds: `verify_email` and `manage`
    - verification path can exchange a token for a short-lived HttpOnly management session
    - management session read/write helpers live in one place so the management UI does not parse raw email-link tokens directly

    Do not reuse the stateless feedback HMAC pattern for creator management.
  </action>
