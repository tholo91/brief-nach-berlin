# Phase 999.6: Landtag + Kommune Politician Coverage Expansion — Research

**Researched:** 2026-05-09
**Domain:** German political data infrastructure — Bund/Land/Kommune routing, data sources, address derivation, and Zuständigkeits-Taxonomie
**Confidence:** HIGH (Bund/Landtag), MEDIUM (Kommune)

---

## Summary

The feature requires three capabilities: (1) a Zuständigkeits-Taxonomie so Mistral can route a user's Anliegen to the correct political level, (2) data about politicians at each level (name, Wahlkreis/Stimmkreis, postal address), and (3) a mapping from a user's PLZ to the right political unit at each level.

**Federal level (Bundestag)** is already implemented. The Abgeordnetenwatch API v2 covers all 16 Bundesländer Landtage fully and is CC0 — this is the primary recommended source for Land-level expansion. No postal address is returned by the API for any level; addresses must be derived from a per-Bundesland building-address formula (parallel to the existing Bundestag formula). PLZ-to-Landtag-Stimmkreis mapping requires per-Bundesland CSV files from the respective statistical offices — there is no single national source equivalent to the Bundeswahlleiterin's Bundestag CSVs.

**Municipal level (Gemeinde)** has no equivalent consolidated open data source. OParl is the best structured standard but covers roughly 100-200 larger cities; smaller Kommunen have no programmatic access. Wikidata SPARQL covers Bürgermeister for larger cities. For v1 expansion, the realistic strategy is: Bürgermeister only (not full Gemeinderäte), sourced from Wikidata, with Rathaus building address derivable from OpenPLZ API (PLZ → Gemeindename → "Rathaus [Gemeindename]").

**Primary recommendation:** Phase in two sub-stages: (A) Landtag via Abgeordnetenwatch — high-effort but high-coverage, fully programmatic; (B) Bürgermeister via Wikidata SPARQL — medium-effort, covers major cities well, acceptable for v1.

**Alternative: Geospatial PLZ-in-Stimmkreis lookup**

**Recommended approach for v1:** Start with the 4 most populous Bundesländer (NRW, Bayern, BW, Niedersachsen — ~50% of Germany's population) using their official CSVs, then expand. Geospatial fallback for states lacking a clean PLZ CSV.

### 1.3 Municipal Level (Gemeinde, Kreis, Stadt)

Municipal-level data has no single national open source. The landscape is fragmented.

#### Option A: OParl API (Ratsinformationssysteme)

**What it is:** Standardized REST+JSON API schema for municipal council information systems (Ratsinformationssysteme). Implemented by commercial RIS vendors (SessionNet, ALLRIS, SD.NET) on behalf of cities.

**Coverage:** Approximately 100-200 German cities have live OParl endpoints [MEDIUM confidence — confirmed implementations in Köln, Bonn, Münster, Düsseldorf, Aachen, and NRW broadly; 22 municipalities confirmed live by 2017 via open.nrw; growth since then uncertain] [CITED: open.nrw/Kommunen-OParl]

**What data is available:** OParl 1.1 schema includes `Person`, `Membership`, `Organization`, `Body`, `Meeting`, `Paper` objects. The `Person` object stores council member names and roles; `Membership` links persons to organizations with `role` and date range fields. [VERIFIED: oparl.org spec]

**What is NOT available in OParl:** Complete list of all registered endpoints (no central registry found). Bürgermeister addresses are not consistently included. [ASSUMED — based on spec review, no confirmed national endpoint registry]

**Practical limitation for Brief nach Berlin v1:** No central endpoint directory means the fetcher would need to maintain a hardcoded list of endpoint URLs. Adding cities requires manual research. Not scalable for a first pass.

**License:** Varies per city — OParl is a standard, not a data license. Each city sets its own terms. [ASSUMED]

**Recommendation:** Use OParl as a secondary, opt-in enrichment source for major cities once the Bürgermeister baseline is established. Not suitable as the primary municipal data path in v1.

#### Option B: Wikidata SPARQL (Bürgermeister)

**What it is:** Wikidata is a CC0 knowledge graph queryable via SPARQL at query.wikidata.org. It has structured data on German Bürgermeister (P39 = "position held", with value = Bürgermeister/Oberbürgermeister + city reference).

**Coverage:** Bürgermeister for Kreisfreie Städte and larger Gemeinden are generally well-covered. Small villages (<5,000 population) have poor Wikidata coverage. [MEDIUM confidence — confirmed via wikidata.org/wiki/WikiProject_every_politician and janakiev.com/blog/wikidata-mayors]

**What data is available:**
- Name, Wikidata ID
- Position held with municipality reference
- Party affiliation (if entered)
- Term dates (quality varies)

**What is NOT in Wikidata:** Postal address. Must be derived as `Rathaus [Gemeindename] / [Rathausadresse]`. The Rathaus address is obtainable via OpenPLZ API (PLZ → Gemeindename) combined with manual lookup or Nominatim geocoding.

**License:** CC0 [VERIFIED: wikidata.org]

**SPARQL query pattern for German mayors:**
```sparql
SELECT ?person ?personLabel ?municipality ?municipalityLabel ?party ?partyLabel WHERE {
  ?person wdt:P39 ?pos .
  ?pos wikibase:directClaim wdt:P39 .
  VALUES ?posType { wd:Q1752346 wd:Q30185 }  # Bürgermeister, Oberbürgermeister
  ?municipality wdt:P1566 ?geonameid .        # German municipalities have GeoNames IDs
  ?municipality wdt:P17 wd:Q183 .             # Germany
  OPTIONAL { ?person wdt:P102 ?party }
  SERVICE wikibase:label { bd:serviceParam wikibase:language "de,en" }
}
```
*Note: Query must be refined to filter for currently active positions. [ASSUMED — query structure confirmed by janakiev.com tutorial, exact filter for active terms requires `wdt:P582` end-date check]*

**Freshness:** Community-edited; Bürgermeisterwechsel may lag reality by weeks to months. Not suitable for time-sensitive accuracy requirements. [ASSUMED — inherent Wikidata characteristic]

**Recommendation:** Use Wikidata as the primary municipal data source for v1, scoped to Kreisfreie Städte and Gemeinden with >20,000 population. Accept coverage gaps for small villages.

#### Option C: Individual Landtag/Kreis Portals (Manual / Scraped)

Some Bundesländer publish structured CSVs or XML feeds of elected officials, including Kreisräte and Gemeinderäte. These are heterogeneous, require per-source maintenance, and are not a v1 priority. [ASSUMED — not individually verified]

#### Municipal Address Formula

Letters to Bürgermeister are addressed to the Rathaus, not the individual's home. The formula is:

```
Herrn / Frau Bürgermeister/in [Name]
Rathaus [Gemeindename]
[Rathausstraße Hausnummer]
[PLZ] [Gemeindename]
```

The Rathaus address can be obtained via:
1. Wikidata (P159 = headquarters location on the municipality item) [ASSUMED — coverage varies]
2. Nominatim geocoding: search "Rathaus [Gemeindename]" returns the building address
3. Manual curation for top-N cities

**Landräte (Kreisebene):** A letter to the Landrat is addressed to the Landratsamt (Kreishaus). Address derivable from Destatis Gemeindeschlüssel + manual lookup. Not recommended for v1 unless a user's Anliegen is clearly Kreis-level (Kreisstraßen, Kreisjugendamt). [ASSUMED]

## Section 2: PLZ Mapping Infrastructure

### 2.1 PLZ → Bundesland (already solved)

The existing `plz-wahlkreis-mapping.json` resolves PLZ → Bundestag Wahlkreis → MdB. Extending to Landtag requires a separate PLZ → Landtag-Stimmkreis mapping per Bundesland.

The PLZ → Bundesland step is available for free via **OpenPLZ API**: [VERIFIED: openplzapi.org]

```
GET https://openplzapi.org/de/Localities?postalCode={plz}
```
Returns: `municipality.key` (Gemeindeschlüssel), `municipality.name`, `municipality.type`, `district.name`, `federalState.key`, `federalState.name`

**Example (80331):** municipality = "München, Landeshauptstadt", district = "München, Landeshauptstadt" (Kreisfreie Stadt), federalState = "Bayern" [VERIFIED: live API call]

This solves:
- Which Bundesland a PLZ is in → know which Landtag to query
- Which Gemeindename to use for Bürgermeister lookup and Rathaus address
- Which Kreis for Landrats-routing (future)

**OpenPLZ API details:**
- URL: https://openplzapi.org
- Free, open data project
- REST/JSON, paginated
- No authentication required
- Germany + Austria + Switzerland + Liechtenstein
- License: not explicitly stated but identified as open data [MEDIUM confidence]

### 2.2 PLZ → Landtag-Stimmkreis

Requires per-Bundesland mapping files. Two approaches:

**Approach 1: CSV from Landesamt (preferred where available)**
Same pattern as existing Bundeswahlleiter CSV processing. Each statistical Landesamt publishes a PLZ-to-Stimmkreis or Gemeinde-to-Stimmkreis table. These need to be located per Bundesland and processed into JSON. Baden-Württemberg confirmed available [CITED: statistik-bw.de/Wahlen/Download_GeoDaten.jsp]. Bayern confirmed available via statistik.bayern.de.

**Approach 2: Geospatial point-in-polygon**
Use PLZ centroid coordinates (from BKG PLZ-Geodaten) + Stimmkreis boundary shapefiles (from each Landesamt) to determine constituency. More robust but more complex build pipeline. Same pattern as Bundeswahlleiter shapefile processing already in the codebase (see `web/data/raw/`).

**Recommended for v1:** Approach 1 for the top-4 Bundesländer (NRW/BY/BW/NI), Approach 2 as fallback. Build one Bundesland at a time; the `PoliticalLevel` type and `politiciansCache.landtag[]` array are already in place.

## Section 3: Zuständigkeits-Taxonomie

This section documents the distribution of political competence so Mistral can route a user's Anliegen to the correct level.

### 3.1 Federal Level (Bund) — Exclusive Competencies (Art. 73 GG)

The Bund has sole authority over:

| Topic Area | Examples of Anliegen | Notes |
|-----------|----------------------|-------|
| Außen- und Verteidigungspolitik | NATO-Beitrag, Bundeswehr-Einsatz, Auslandshilfe | Pure Bund |
| Staatsbürgerschaft & Pass | Einbürgerung, Ausweis, Grenzkontrollen | Pure Bund |
| Währung & Bundesfinanzen | Euro-Stabilität, Bundeshaushalt | Pure Bund (ECB is EU, but Bundestag role here) |
| Luftverkehr | Fluglärm am Flughafen, Billigflug-Regulierung | Pure Bund (Luftrecht) |
| Eisenbahn des Bundes | Deutsche Bahn Zugverspätungen, Streckenstilllegungen | Pure Bund |
| Post und Telekommunikation | Breitbandausbau (Bundesförderung), Postgesetz | Bund; implementation varies |
| Urheberrecht & gewerblicher Rechtsschutz | Streaming-Lizenzen, Softwarepatente | Pure Bund |
| Kernenergie & Strahlenschutz | AKW-Laufzeiten, Atommüll-Endlager | Pure Bund |
| Waffen & Sprengstoff | Waffengesetz, Schusswaffenverbote | Pure Bund |
| Terrorismusbekämpfung (federal) | BKA, Verfassungsschutz | Pure Bund |
| Sozialversicherung | Rentenversicherung, Krankenversicherung (GKV), Pflegeversicherung, ALG | Bund (SGB) |

[VERIFIED: Art. 73 GG via grundgesetz-fuer-jeden.de; bpb.de/themen/politisches-system/]

### 3.2 Concurrent Competencies (Art. 74 GG) — Bund dominates framework

Bund sets law; Länder may implement but cannot contradict:

| Topic Area | Practical Routing | Examples of Anliegen |
|-----------|-------------------|---------------------|
| Strafrecht & Strafvollzug | Bund (Strafrecht), Land (Strafvollzug) | Gefängnisreform, Resozialisierung |
| Wirtschaftsrecht | Bund | Marktregulierung, Kartellrecht |
| Arbeitsrecht | Bund | Mindestlohn, Kündigungsschutz, Arbeitszeitgesetz |
| Umweltschutz & Klimaschutz (Bundesebene) | Bund (Rahmen), Land (Umsetzung) | CO2-Steuer, Klimaschutzgesetz |
| Ausländerrecht & Asyl | Bund | Aufenthaltsrecht, Asylverfahren |
| Sozialhilfe & Wohngeld | Bund (Gesetz), Land/Kommune (Auszahlung) | Bürgergeld, Wohngeld, Grundsicherung |
| Straßenverkehr & KFZ | Bund | StVO, Führerscheinrecht, Tempolimit |
| Lebensmittelrecht | Bund | Lebensmittelkennzeichnung, Bio-Siegel |
| Gesundheitsrecht (Rahmen) | Bund | Krankenhaus-Finanzierungsgesetz, Impfpflicht |
| Wohnungsbaurecht (Rahmen) | Bund | Mietpreisbremse, BauGB |
| Hochschulzulassung (Rahmen) | Bund/Land | NC-Reform, Studiengebühren |

[VERIFIED: Art. 74 GG via grundgesetz-fuer-jeden.de]

### 3.3 State Level (Länder) — Exclusive Competencies

Länder govern everything not explicitly assigned to the Bund (Art. 70 GG):

| Topic Area | Examples of Anliegen | Notes |
|-----------|----------------------|-------|
| Bildung & Schulen | Lehrermangel, Schulschließungen, Lehrplan, G8/G9 | Kultushoheit der Länder — strongest Land-exclusive area |
| Hochschulen (konkret) | Studiengebühren (Landesrecht), Uni-Finanzierung | Land |
| Polizei & innere Sicherheit | Polizeipräsenz, Polizeigesetze, Videoüberwachung | Ländersache |
| Verfassungsschutz (Landes-) | Landesverfassungsschutz | Land |
| Strafvollzug | Gefängnisreformen, Knastzustände | Land (seit 2006 Föderalismusreform) |
| Rundfunk & Medien | ARD/ZDF-Beitrag, öffentlich-rechtlicher Rundfunk | Ländersache (Rundfunkstaatsvertrag) |
| Gesundheitsversorgung (Umsetzung) | Krankenhausinvestitionen, Notarztversorgung | Land (Krankenhausplanung) |
| Wohnungsbau (konkret) | Sozialer Wohnungsbau im Land, Landesbauordnung | Land |
| Beamtenrecht (Landesbeamte) | Lehrergehälter, Polizistenbesoldung | Land |
| Versammlungsrecht | Demonstrationsrecht (nach Föderalismusreform) | Land |
| Ladenöffnungszeiten | Sonntagsöffnung, Ladenschluss | Land (seit 2006) |
| Gaststättenrecht | Rauchverbote, Öffnungszeiten | Land |
| Heimrecht / Heimgesetz | Altenpflegeheime, Seniorenheime | Land |
| Naturschutz (konkret) | Biotopabriss, Windkraft-Abstandsregeln | Abweichungsrecht der Länder |
| Jagd | Jagdzeiten, Schonzeiten | Länder haben Abweichungsrecht |

[VERIFIED: bpb.de, Art. 70/74 GG, Föderalismusreform 2006 confirmed via bpb.de search]

### 3.4 Municipal Level (Gemeinde) — Kommunale Selbstverwaltung (Art. 28 GG)

Gemeinden have "Allzuständigkeit" (general competence) for all local affairs under Art. 28 Abs. 2 GG. They exercise either Pflichtaufgaben (mandatory) or freiwillige Aufgaben (discretionary):

| Topic Area | Examples of Anliegen | Type |
|-----------|----------------------|------|
| Lokale Infrastruktur & Gemeindestraßen | Schlaglöcher, fehlende Radwege, Straßenbeleuchtung | Pflicht |
| Wasser & Abwasser | Leitungswasser-Qualität, Kanalarbeiten | Pflicht |
| Abfallentsorgung | Mülleimer, Sperrmüll-Abholung | Pflicht |
| Kindergärten & Kitas | Kitaplätze, Kita-Öffnungszeiten | Pflicht (KiföG) |
| Grundschulen (Trägerschaft) | Schulgebäude-Zustand, Schulhof | Pflicht (Träger, nicht Lehrer) |
| Bauanträge & Bauleitplanung | Nachbar baut ohne Genehmigung, Bebauungsplan | Pflicht |
| Öffentlicher Nahverkehr (ÖPNV lokal) | Buslinie wird gestrichen, Takt wird reduziert | Pflicht/Pflicht je nach Kreisstruktur |
| Parks & Grünanlagen | Park wird bebaut, Spielplatz veraltet | Freiwillig |
| Kultureinrichtungen | Bibliothek schließt, Stadttheater-Budget | Freiwillig |
| Sportstätten | Schwimmbad schließt, Sporthalle Zustand | Freiwillig |
| Ortsgestaltung & Sauberkeit | Graffiti, Vandalismus, Stadtreinigung | Pflicht/Freiwillig |
| Gewerbezulassung lokal | Lärm durch Gewerbe, Baugenehmigung | Pflicht |
| Sozialdienste lokal | Obdachlosenunterkunft, Suchtberatung | Pflicht (delegiert) |

[VERIFIED: kommunalbrevier.de, Art. 28 GG, bpb.de research]

**Important nuance:** Many topics span levels. Example: "Wohnungsbau" — Mietpreisbremse is Bund, sozialer Wohnungsbau is Land, Bebauungsplan ist Gemeinde. Mistral must handle this ambiguity and route to the most local effective actor.

### 3.5 Taxonomy: Ambiguous / Multi-Level Topics

These require Mistral to pick the most actionable level based on the specific Anliegen:

| Topic | Bund | Land | Gemeinde |
|-------|------|------|---------|
| Wohnungsbau | Mietrecht, BauGB | Sozialer Wohnungsbau, Landesbauordnung | Bebauungsplan, Baugenehmigung |
| Bildung | Bafög, Hochschulrahmen | Schulen, Unis | Schulträger (Gebäude) |
| Umwelt | CO2-Gesetz, Klimaschutzgesetz | Naturschutzgebiete, Windenergieabstände | Bebauungsplan, Parkflächen |
| Gesundheit | GKV-Reform, Krankenhausgesetz | Krankenhausplanung, Landesärztekammer | Keine direkte Zuständigkeit |
| ÖPNV | Fernverkehr (DB) | Nahverkehrsgesetz (Rahmen) | Buslinien, U-Bahn in Städten |
| Integration | Asylgesetz | Unterbringung, Sprachkurse (Land) | Unterkünfte, Sozialberatung |

### 3.6 Recommended Implementation: Taxonomie als JSON

Store the routing taxonomy as a static JSON file at build time, consumed by the Mistral system prompt:

```json
{
  "bund": {
    "exclusive": ["Außenpolitik", "Bundeswehr", "Sozialversicherung (GKV/Rente/Pflege)", "Bahn (DB)", "Straßenverkehrsrecht", "Telekommunikation", "Asylrecht", "Staatsangehörigkeit"],
    "concurrent_dominant": ["Arbeitsrecht", "Umweltrahmen", "Wohnungsbaurecht (Mietpreisbremse)", "Wirtschaftsrecht"]
  },
  "land": {
    "exclusive": ["Schulen und Bildung", "Polizei", "Strafvollzug", "Rundfunk", "Krankenhausplanung", "Sozialer Wohnungsbau", "Ladenöffnungszeiten", "Versammlungsrecht"]
  },
  "gemeinde": {
    "exclusive": ["Lokale Straßen", "Kitas (Träger)", "Bebauungsplan", "Abfallentsorgung", "ÖPNV lokal", "Grünanlagen", "Schwimmbäder", "Bibliotheken"]
  }
}
```

This taxonomy is fed as part of the Mistral routing prompt so Mistral can reason: "Anliegen mentions Schlaglöcher → lokale Straße → Gemeinde → Bürgermeister."

**OpenPLZ API quality assessment:** Medium-high confidence. Live API confirmed working (PLZ 80331 tested). Returns Gemeindename, Gemeindeschlüssel, Bundesland. Active project maintained on GitHub. No authentication. Ideal for runtime PLZ → Gemeinde resolution.

### Pattern: Mistral Level-Routing Call (New)

**What:** A lightweight Mistral call that classifies the Anliegen against the taxonomy before the main letter-generation call.

**When to use:** Before politician lookup. Returns `level` + `confidence` + `reasoning`. If confidence is low, show the user a "Welche Ebene betrifft das?" disambiguation step.

**Example (server action):**
```typescript
// Source: architecture design — no external docs reference
interface RoutingResult {
  level: PoliticalLevel;
  confidence: "high" | "medium" | "low";
  reasoning: string;
}

async function routeToLevel(anliegen: string): Promise<RoutingResult> {
  const response = await mistral.chat.complete({
    model: "mistral-small-latest",  // routing is cheap; use smaller model
    temperature: 0.1,
    messages: [{
      role: "system",
      content: `Du bist ein Experte im deutschen Föderalismus. Ordne das folgende Anliegen einer politischen Ebene zu.
      
Ebenen:
- Bund: ${BUND_TOPICS.join(", ")}
- Land: ${LAND_TOPICS.join(", ")}
- Gemeinde: ${GEMEINDE_TOPICS.join(", ")}

Antworte als JSON: { "level": "Bund"|"Land"|"Gemeinde", "confidence": "high"|"medium"|"low", "reasoning": "..." }`
    }, {
      role: "user",
      content: `Anliegen: ${anliegen}`
    }]
  });
  return JSON.parse(response.choices[0].message.content);
}
```

**Cost:** ~50-100 tokens per call on mistral-small. Negligible.

### Anti-Patterns to Avoid

- **Hand-rolling PLZ → Gemeinde logic:** OpenPLZ API solves this cleanly.
- **Building a central Gemeinderatsmitglieder DB in v1:** Out of scope; OParl complexity > value at this stage.
- **Hardcoding Landtag period IDs:** These change each election. Resolve dynamically via `parliament-periods?parliament={id}&type=legislature` sorted by `start_date_period` desc.
- **Assuming Stimmkreis IDs are stable across elections:** They are NOT — same issue as Bundestag Wahlkreis IDs. Constituency IDs differ per parliament_period in Abgeordnetenwatch. [VERIFIED: AW API docs note this explicitly]

## Common Pitfalls

### Pitfall 1: Constituency IDs Change Per Election
**What goes wrong:** The build script caches a constituency ID that becomes invalid after the next election.
**Why it happens:** Abgeordnetenwatch creates new constituency objects for each parliament_period. The same geographic Wahlkreis has a different ID in the 2021 and 2025 periods.
**How to avoid:** Always resolve the current period ID at build time via `parliament-periods?parliament={id}` sorted by `start_date_period` desc; never hardcode period or constituency IDs.
**Warning signs:** Candidacy-mandate queries returning 0 results despite a valid parliament ID.

### Pitfall 2: PLZ Spans Multiple Bundesländer
**What goes wrong:** A PLZ is assumed to be entirely within one Bundesland, but some border PLZs span two Länder.
**Why it happens:** PLZ boundaries are Deutsche Post administrative zones, not political boundaries.
**How to avoid:** OpenPLZ API returns multiple results for such PLZs. Collect all, determine the dominant Bundesland by municipality count. Existing disambiguation UI already handles this pattern.
**Warning signs:** OpenPLZ returning multiple `federalState` values for the same PLZ.

### Pitfall 3: Abgeordnetenwatch Has No Postal Address
**What goes wrong:** Build script tries to use an AW API field for postal address that doesn't exist.
**Why it happens:** AW deliberately excludes personal contact details from the API.
**How to avoid:** Always derive postal address from the Landtag building-address formula. Store formulas in `landtag-addresses.json` keyed by Bundesland abbreviation.
**Warning signs:** Empty address field in politician cache.

### Pitfall 4: Wikidata Bürgermeister Data Staleness
**What goes wrong:** Generated letter goes to a Bürgermeister who left office 6 months ago.
**Why it happens:** Wikidata is community-edited; small-city updates lag by months.
**How to avoid:** Add a disclaimer in the letter-generation UI for the Gemeinde level: "Bitte prüfe vor dem Absenden, ob [Name] noch im Amt ist." Regenerate Wikidata cache monthly.
**Warning signs:** Bürgermeister name mismatch if user notices.

### Pitfall 5: Bayern Has 207 Stimmkreise, Not 17 Wahlkreise
**What goes wrong:** Developer conflates Bundestag Wahlkreise (299 nationwide) with Bayern Stimmkreise (207 within Bayern).
**Why it happens:** Bayern calls its state-level constituencies "Stimmkreise" not "Wahlkreise". Other Bundesländer use "Wahlkreis" for both levels with context.
**How to avoid:** Use the term from each Bundesland's own documentation. Abgeordnetenwatch normalizes these as `constituency` regardless of local terminology.

## Open Questions

1. **Stimmkreis CSV availability per Bundesland**
   - What we know: BW confirmed, BY confirmed directory exists. 13 others not individually verified.
   - What's unclear: Whether all 16 Landesämter publish a PLZ-level CSV (vs. only Gemeinde-level, which requires a Gemeinde→PLZ join).
   - Recommendation: Investigate each Landesamt before build sprint. Fallback: geospatial lookup using BKG PLZ centroids + Stimmkreis shapefiles.

2. **Wikidata coverage threshold for Bürgermeister**
   - What we know: Larger cities well covered; small villages not.
   - What's unclear: Exact cutoff. "20,000 population" is an educated estimate.
   - Recommendation: Run the SPARQL query on the build machine in a spike; count municipalities returned vs. total ~11,000 German municipalities. Accept <50% coverage for v1.

3. **Rathaus postal address sourcing for Wikidata municipalities**
   - What we know: Wikidata has P159 (headquarters). Nominatim search works.
   - What's unclear: P159 coverage completeness.
   - Recommendation: Use Nominatim as primary (`search?q=Rathaus+{GemeindeName}&format=json`), P159 as fallback. Cache at build time.

4. **Abgeordnetenwatch rate limits**
   - What we know: No documented rate limit. Existing Bundestag fetch works without issues.
   - What's unclear: Whether fetching all 16 Landtage simultaneously triggers throttling.
   - Recommendation: Sequential fetch with 200ms delay between parliament periods. Monitor HTTP 429 responses.
